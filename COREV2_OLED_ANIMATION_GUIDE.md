## 1. Quick Start: Wokwi Animator

### What It Is

[Wokwi Animator](https://animator.wokwi.com) is a free browser tool that converts **Icons8 animated icons** into C/C++ bitmap arrays ready for Arduino/ESP32 OLED displays. It handles the hard work of converting GIF frames into PROGMEM byte arrays.

### Step-by-Step Workflow

```
1. Open https://animator.wokwi.com
2. Browse animation categories:
   ┌──────────┬──────────┬──────────┐
   │ UI       │ Web      │ Social   │
   │ Media    │ Weather  │ Ecommerce│
   │ Loaders  │ Emoji    │ Gestures │
   │ Food     │ Sports   │          │
   └──────────┴──────────┴──────────┘
3. Click an animation to select it
4. Choose SIZE from the dropdown:
   • 32×32 — small icons, good for HUD elements
   • 48×48 — medium, balanced detail vs memory
   • 64×64 — large, most detailed (use sparingly!)
5. Click "GET THE CODE >"
6. Copy the generated C code into your project
```

### What the Generated Code Looks Like

```cpp
// Generated by Wokwi Animator
// Animation: heart-beat, 32×32, 20 frames

#include <avr/pgmspace.h>  // For AVR. Use this for ESP32 compatibility:
// #define PROGMEM  // (ESP32 ignores PROGMEM — still use for portability)

const unsigned char animation_heart_beat[20][128] PROGMEM = {
  // Frame 0
  { 0x00,0x00,0x00,0x00, 0x00,0x00,0x00,0x00, 0x00,0x00,0x00,0x00,
    0x00,0x0F,0xF0,0x00, 0x00,0x3F,0xFC,0x00, 0x00,0x7F,0xFE,0x00,
    0x00,0xFF,0xFF,0x00, 0x01,0xFF,0xFF,0x80, 0x01,0xFF,0xFF,0x80,
    0x01,0xFF,0xFF,0x80, 0x01,0xFF,0xFF,0x80, 0x00,0xFF,0xFF,0x00,
    0x00,0x7F,0xFE,0x00, 0x00,0x3F,0xFC,0x00, 0x00,0x0F,0xF0,0x00,
    /* ... rest of frame ... */ },
  // Frame 1
  { /* ... */ },
  // ... up to frame 19
};
```

### Key Points About Generated Code

| Detail | Explanation |
|--------|-------------|
| **PROGMEM** | Tells the compiler to store in flash, not RAM. ESP32 ignores this keyword but it's harmless to keep. |
| **Array dimensions** | `[frameCount][bytesPerFrame]` — `bytesPerFrame = ceil(width × height / 8)` |
| **Byte order** | Each byte represents 8 vertical pixels, MSB at top. Next byte is next column to the right. |
| **32×32 image** | `ceil(32×32/8) = 128` bytes per frame |
| **48×48 image** | `ceil(48×48/8) = 288` bytes per frame |
| **64×64 image** | `ceil(64×64/8) = 512` bytes per frame |

## 4. Understanding the Bitmap Format

### Pixel-to-Byte Mapping

The SSD1306/SH1106 uses a **vertical byte orientation**. Each byte encodes 8 pixels in a column, with the LSB at the top:

```
Byte: 0b11000110
       ││││││││
       ▼▼▼▼▼▼▼▼
Pixel 0: 0 ──□ (white/off)
Pixel 1: 1 ──■ (black/on)  
Pixel 2: 1 ──■
Pixel 3: 0 ──□
Pixel 4: 0 ──□
Pixel 5: 0 ──□
Pixel 6: 1 ──■
Pixel 7: 1 ──■
```

### How Adafruit GFX `drawBitmap()` Works

```cpp
// drawBitmap(x, y, bitmap, w, h, color)
// x, y: top-left corner position
// bitmap: pointer to PROGMEM byte array
// w, h: width and height in pixels
// color: SSD1306_WHITE (pixel on) or SSD1306_BLACK (pixel off / erase)

display.drawBitmap(48, 16,                    // Draw at center-ish of 128×64
                   animation_heart_beat[0],   // First frame
                   32, 32,                    // 32×32 pixels
                   SSD1306_WHITE);
```

### Manual Bitmap Creation (without Wokwi)

When you need a custom icon that doesn't exist in Wokwi/Icons8, create the bitmap manually:

```cpp
// A simple 8×8 heart icon, hand-drawn
// Use binary literal syntax for readability
const unsigned char PROGMEM icon_heart_8x8[] = {
  0b01100110,  //   ■■  ■■
  0b11111111,  //  ■■■■■■■■
  0b11111111,  //  ■■■■■■■■
  0b11111111,  //  ■■■■■■■■
  0b01111110,  //   ■■■■■■
  0b00111100,  //    ■■■■
  0b00011000,  //     ■■
  0b00000000,  //
};
// Usage: display.drawBitmap(x, y, icon_heart_8x8, 8, 8, SSD1306_WHITE);
```

### Bitmap Design Tips

1. **Draw on graph paper first**, then convert to binary
2. **Use comment annotations** with █ and space to visualize each byte
3. **Online tools** (besides Wokwi) that help:
   - [image2cpp (javl.github.io)](https://javl.github.io/image2cpp/) — convert PNG to byte array
   - [LCD Image Converter](https://lcd-image-converter.github.io/) — desktop tool
   - [dotmatrixtool.com](https://dotmatrixtool.com) — draw pixel art, export as C array

## 5. Rendering Animations on the CoreV2 OLED

### The Animation Loop Pattern

Every animation follows this structure:

```cpp
#include <Adafruit_GFX.h>
#include <Adafruit_SSD1306.h>

// ── Display Setup (CoreV2 specific) ──────────────────────────
#define SCREEN_WIDTH  128
#define SCREEN_HEIGHT 64
#define OLED_ADDR     0x3C
#define I2C_SDA       8
#define I2C_SCL       9

Adafruit_SSD1306 display(SCREEN_WIDTH, SCREEN_HEIGHT, &Wire, -1);

// ── Animation Data (from Wokwi Animator) ─────────────────────
// Replace this with your generated code
const unsigned char anim_example[10][128] PROGMEM = { /* ... */ };
const uint8_t ANIM_FRAMES = 10;
const uint8_t ANIM_WIDTH  = 32;
const uint8_t ANIM_HEIGHT = 32;
// bytes per frame = ceil(ANIM_WIDTH * ANIM_HEIGHT / 8)
const uint16_t ANIM_BYTES = (ANIM_WIDTH * ANIM_HEIGHT) / 8;

// ── Animation State ──────────────────────────────────────────
uint8_t  currentFrame = 0;
uint32_t lastFrameTime = 0;
const uint32_t FRAME_DELAY_MS = 80;  // ~12.5 FPS

// ── Animation Position ───────────────────────────────────────
// Center a 32×32 animation on 128×64 screen
const int16_t ANIM_X = (SCREEN_WIDTH  - ANIM_WIDTH)  / 2;   // (128-32)/2 = 48
const int16_t ANIM_Y = (SCREEN_HEIGHT - ANIM_HEIGHT) / 2;   // (64-32)/2  = 16

void setup() {
  Wire.begin(I2C_SDA, I2C_SCL);
  display.begin(SSD1306_SWITCHCAPVCC, OLED_ADDR);
  display.clearDisplay();
}

void loop() {
  // ── Frame Timing ────────────────────────────────────────
  uint32_t now = millis();
  if (now - lastFrameTime >= FRAME_DELAY_MS) {
    lastFrameTime = now;

    // ── Clear previous frame ──────────────────────────────
    display.clearDisplay();

    // ── Draw current frame ────────────────────────────────
    display.drawBitmap(
      ANIM_X, ANIM_Y,
      anim_example[currentFrame],
      ANIM_WIDTH, ANIM_HEIGHT,
      SSD1306_WHITE
    );

    // ── Push to display ───────────────────────────────────
    display.display();

    // ── Advance frame (loop) ──────────────────────────────
    currentFrame++;
    if (currentFrame >= ANIM_FRAMES) {
      currentFrame = 0;  // Loop back to start
    }
  }
}
```

### Frame Timing Rules

| FPS | Frame Delay | Best For |
|-----|-------------|----------|
| 30 fps | 33 ms | Very smooth motion, small sprites only |
| 15 fps | 66 ms | Smooth — character animations, bouncing icons |
| **12 fps** | **83 ms** | **Sweet spot for 128×64 OLED** — smooth enough, CPU-friendly |
| 10 fps | 100 ms | Good — loading spinners, status indicators |
| 6 fps | 166 ms | Acceptable — weather icons, simple state changes |
| 3 fps | 333 ms | Minimal — slow transitions, battery indicators |

### Clearing Only the Animation Area (Performance Hack)

Instead of `display.clearDisplay()` (which clears all 128×64 pixels), only clear the animation region:

```cpp
// Fill just the animation rectangle with black
display.fillRect(ANIM_X, ANIM_Y, ANIM_WIDTH, ANIM_HEIGHT, SSD1306_BLACK);
display.drawBitmap(ANIM_X, ANIM_Y, anim_example[currentFrame],
                   ANIM_WIDTH, ANIM_HEIGHT, SSD1306_WHITE);
display.display();
```

This is **3–5× faster** on I²C displays and prevents flicker of static UI elements (HUD, text, etc.).

## 6. Animation Pattern Catalog

### 4.1 Bouncing Icon

An icon that moves around the screen, bouncing off edges. Great for idle screens.

```cpp
// ── Bouncing Heart (32×32) ──────────────────────────────────

float bx = 48.0f, by = 16.0f;        // Position
float bvx = 1.2f, bvy = 0.8f;        // Velocity
const float GRAVITY = 0.05f;          // Optional: downward pull
const float BOUNCE = 0.7f;            // Energy retained on bounce

void updateBounce() {
  // Apply velocity
  bx += bvx;
  by += bvy;
  // bvy += GRAVITY;  // Uncomment for gravity bounce

  // Bounce off edges
  if (bx <= 0 || bx >= SCREEN_WIDTH - ANIM_WIDTH) {
    bvx = -bvx * BOUNCE;
    bx = constrain(bx, 0, SCREEN_WIDTH - ANIM_WIDTH);
  }
  if (by <= 0 || by >= SCREEN_HEIGHT - ANIM_HEIGHT) {
    bvy = -bvy * BOUNCE;
    by = constrain(by, 0, SCREEN_HEIGHT - ANIM_HEIGHT);
  }
}

void drawBounce() {
  display.fillRect((int)bx, (int)by, ANIM_WIDTH, ANIM_HEIGHT, SSD1306_BLACK);
  display.drawBitmap((int)bx, (int)by,
                     anim_example[currentFrame],
                     ANIM_WIDTH, ANIM_HEIGHT, SSD1306_WHITE);
}
```

### 4.2 Loading Spinner

A rotating indicator. Use the "Loaders" category in Wokwi Animator.

```cpp
// ── Spinner (procedural — no bitmap needed!) ────────────────
void drawSpinner(int16_t cx, int16_t cy, uint8_t radius, uint8_t tick) {
  // Draw 8 dots around a circle, highlight the active one
  for (uint8_t i = 0; i < 8; i++) {
    float angle = i * PI / 4.0f;  // 45° spacing
    int16_t dx = cos(angle) * radius;
    int16_t dy = sin(angle) * radius;
    if (i == tick % 8) {
      display.fillCircle(cx + dx, cy + dy, 2, SSD1306_WHITE);  // Active dot
    } else {
      display.drawCircle(cx + dx, cy + dy, 2, SSD1306_WHITE);  // Outline dot
    }
  }
}
// Call with: drawSpinner(64, 32, 12, frameCounter / 4);
```

### 4.4 Progress Bar

Essential for Smart Safe (lock progress) and Smart Plant Pot (moisture level).

```cpp
void drawProgressBar(int16_t x, int16_t y, int16_t w, int16_t h,
                     uint8_t percent) {
  // Outline
  display.drawRect(x, y, w, h, SSD1306_WHITE);
  // Fill proportional to percent (0–100)
  int16_t fillW = (w - 2) * percent / 100;
  if (fillW > 0) {
    display.fillRect(x + 1, y + 1, fillW, h - 2, SSD1306_WHITE);
  }
}

// Animated fill example
void animateProgressBar(uint8_t targetPercent) {
  static uint8_t currentPercent = 0;
  if (currentPercent < targetPercent) currentPercent++;
  if (currentPercent > targetPercent) currentPercent--;
  drawProgressBar(14, 48, 100, 10, currentPercent);
}
```

### 4.5 Blinking / Pulsing Effect

```cpp
// ── Pulsing LED indicator ───────────────────────────────────
void drawPulsingDot(int16_t cx, int16_t cy, uint32_t periodMs) {
  // Sine-wave brightness via PWM-style dithering
  float phase = (millis() % periodMs) / (float)periodMs;  // 0.0–1.0
  float brightness = (sin(phase * 2 * PI) + 1.0f) / 2.0f;  // 0.0–1.0

  // On monochrome OLED, simulate brightness with dither pattern
  uint8_t radius = 1 + (uint8_t)(brightness * 3);  // 1–4 pixel radius
  display.fillCircle(cx, cy, radius, SSD1306_WHITE);
}
```

```cpp
void drawScrollingText(const char* text, int16_t y, int16_t* offsetX) {
  int16_t x = SCREEN_WIDTH - *offsetX;
```

### 4.8 Confetti / Particle Burst

Great for "unlock" celebration (Smart Safe) or "score reached" (Game Boy).

```cpp
// ── Particle burst system ───────────────────────────────────
struct Particle {
  float x, y, vx, vy;
  uint8_t life;  // Frames remaining
};

#define MAX_PARTICLES 30
Particle particles[MAX_PARTICLES];

void spawnBurst(int16_t cx, int16_t cy) {
  for (uint8_t i = 0; i < MAX_PARTICLES; i++) {
    particles[i].x = cx;
    particles[i].y = cy;
    float angle = random(0, 360) * PI / 180.0f;
    float speed = random(10, 40) / 10.0f;
    particles[i].vx = cos(angle) * speed;
    particles[i].vy = sin(angle) * speed;
    particles[i].life = random(15, 40);
  }
}

void updateAndDrawParticles() {
  for (uint8_t i = 0; i < MAX_PARTICLES; i++) {
    if (particles[i].life > 0) {
      particles[i].x += particles[i].vx;
      particles[i].y += particles[i].vy;
      particles[i].vy += 0.1f;  // Gravity
      particles[i].life--;
      display.drawPixel((int)particles[i].x, (int)particles[i].y,
                        SSD1306_WHITE);
    }
  }
}
```

## 7. Icon Sprite Sheets

### Concept

A sprite sheet packs multiple icon frames into a **single bitmap**, then uses `drawBitmap()` with an offset to pick the right sub-image. This is more memory-efficient than separate arrays.

### Example: 4-Icon Sprite Sheet (each 16×16, total 64×16)

```cpp
// Sprite sheet: [Icon0][Icon1][Icon2][Icon3]  —  each 16×16
const unsigned char PROGMEM sprite_sheet_icons[128] = {
  /* Icon 0: Lock (locked)   */  0x00,0x00, /* ... 32 bytes */ 0x00,0x00,
  /* Icon 1: Lock (unlocked) */  0x00,0x00, /* ... 32 bytes */ 0x00,0x00,
  /* Icon 2: Key             */  0x00,0x00, /* ... 32 bytes */ 0x00,0x00,
  /* Icon 3: Shield          */  0x00,0x00, /* ... 32 bytes */ 0x00,0x00,
};

// Draw icon by index (0–3), 16×16 each
void drawIcon16(uint8_t index, int16_t x, int16_t y) {
  // Calculate byte offset into the sprite sheet
  // Each 16×16 icon = ceil(16*16/8) = 32 bytes
  const uint8_t* iconPtr = &sprite_sheet_icons[index * 32];
  display.drawBitmap(x, y, iconPtr, 16, 16, SSD1306_WHITE);
}
```

### Sprite Sheet Memory Formula

```
bytesPerIcon = ceil(iconWidth × iconHeight / 8)
totalBytes   = bytesPerIcon × numberOfIcons
```

| Icon Size | Bytes per Icon | 4 Icons | 8 Icons | 16 Icons |
|-----------|---------------|---------|---------|----------|
| 8×8 | 8 | 32 B | 64 B | 128 B |
| 16×16 | 32 | 128 B | 256 B | 512 B |
| 24×24 | 72 | 288 B | 576 B | 1.1 KB |
| 32×32 | 128 | 512 B | 1 KB | 2 KB |
| 48×48 | 288 | 1.1 KB | 2.3 KB | 4.5 KB |
| 64×64 | 512 | 2 KB | 4 KB | 8 KB |

> **CoreV2 rule:** With 16 MB flash, sprite sheets are never a problem. Prefer them for organization.

## 8. Procedural Animations (No Bitmaps)

Procedural animations use math + drawing primitives instead of pre-made bitmap arrays. They use **zero flash storage** (only code) and can be infinitely varied.

### 6.1 Sine Wave Oscillator

```cpp
void drawSineWave(int16_t baseY, float amplitude, float frequency,
                  uint32_t timeMs) {
  int16_t prevY = baseY;
  for (int16_t x = 0; x < SCREEN_WIDTH; x++) {
    float phase = (x / (float)SCREEN_WIDTH) * 2 * PI * frequency;
    phase += timeMs / 200.0f;  // Animate over time
    int16_t y = baseY + (int16_t)(sin(phase) * amplitude);
    if (x > 0) {
      display.drawLine(x - 1, prevY, x, y, SSD1306_WHITE);
    }
    prevY = y;
  }
}
```

### 6.2 Analog Clock Face

```cpp
void drawClock(int16_t cx, int16_t cy, uint8_t radius,
               uint8_t hour, uint8_t minute) {
  // Draw circle outline
  display.drawCircle(cx, cy, radius, SSD1306_WHITE);

  // Hour hand (shorter, thicker)
  float hAngle = (hour % 12) * 30.0f + minute * 0.5f;  // degrees
  hAngle = hAngle * PI / 180.0f - PI / 2.0f;
  int16_t hx = cx + cos(hAngle) * radius * 0.5f;
  int16_t hy = cy + sin(hAngle) * radius * 0.5f;
  display.drawLine(cx, cy, hx, hy, SSD1306_WHITE);

  // Minute hand (longer)
  float mAngle = minute * 6.0f;
  mAngle = mAngle * PI / 180.0f - PI / 2.0f;
  int16_t mx = cx + cos(mAngle) * radius * 0.8f;
  int16_t my = cy + sin(mAngle) * radius * 0.8f;
  display.drawLine(cx, cy, mx, my, SSD1306_WHITE);
}
```

### 6.3 Starfield (Parallax Scrolling Background)

```cpp
#define MAX_STARS 40
struct Star { float x, y; uint8_t speed; };  // speed 1=slow, 3=fast
Star stars[MAX_STARS];

void initStarfield() {
  for (uint8_t i = 0; i < MAX_STARS; i++) {
    stars[i].x = random(SCREEN_WIDTH);
    stars[i].y = random(SCREEN_HEIGHT);
    stars[i].speed = random(1, 4);  // 1, 2, or 3
  }
}

void updateStarfield() {
  for (uint8_t i = 0; i < MAX_STARS; i++) {
    stars[i].x -= stars[i].speed * 0.3f;  // Move left at different speeds
    if (stars[i].x < 0) {
      stars[i].x = SCREEN_WIDTH;
      stars[i].y = random(SCREEN_HEIGHT);
    }
    display.drawPixel((int)stars[i].x, (int)stars[i].y, SSD1306_WHITE);
  }
}
```

```cpp
    drops[i].y = random(SCREEN_HEIGHT);

void updateRain() {

    drops[i].y += drops[i].speed;

      drops[i].x = random(SCREEN_WIDTH);

    display.drawPixel((int)drops[i].x, (int)drops[i].y - 1, SSD1306_WHITE);
  }
}
```

## 9. Performance & Memory Guidelines

### The Golden Rules for CoreV2 (ESP32-S3, 128×64 OLED, I²C)

| Rule | Why |
|------|-----|
| **Limit animation area to ≤ 64×64** | Larger than 64×64 on a 128-wide screen causes visible tearing on I²C |
| **Target 10–15 FPS** | I²C at 400 kHz can push ~15 fps for 64×64 area. Don't try 30 fps — the bus can't keep up. |
| **Use `fillRect` not `clearDisplay`** | Partial clears are 3–5× faster. Only `clearDisplay()` on full scene changes. |
| **PROGMEM for bitmaps** | Always. With 16 MB flash you have room, but PROGMEM is good practice. |
| **Inline small icons** | Icons ≤ 16×16 can be `const` in the function scope. Larger ones deserve their own file. |
| **Don't chain `display.display()` calls** | Each `display()` pushes the full 1 KB framebuffer over I²C. Batch all drawing, then call `display()` once. |
| **Precompute animation positions** | Don't do float math in the draw loop if you can precompute. Use `uint16_t` look-up tables for sine/cosine. |

### Memory Budget per Animation

| Animation Type | Flash (PROGMEM) | RAM | I²C Bus Time |
|---------------|-----------------|-----|-------------|
| 32×32 × 10 frames | 1.25 KB | ~0 B | ~10 ms/frame |
| 48×48 × 10 frames | 2.8 KB | ~0 B | ~20 ms/frame |
| 64×64 × 10 frames | 5 KB | ~0 B | ~35 ms/frame |
| Procedural (code only) | ~200 B code | ~100 B state | Varies |
| Sprite sheet 16×16 × 8 icons | 256 B | ~0 B | ~2 ms/icon |

### Frame Rate Calculator

```
maxFPS ≈ 1 / (I2C_transfer_time + draw_time)

Where:
  I2C_transfer_time = (bytes_to_send × 9 bits) / I2C_clock
  At 400 kHz: (1024 bytes × 9) / 400000 ≈ 23 ms per full frame

  Partial update: (area_bytes × 9) / 400000
  32×32 area: (128 bytes × 9) / 400000 ≈ 2.9 ms
  64×64 area: (512 bytes × 9) / 400000 ≈ 11.5 ms
```

## 11. Custom Animation Workflow

### Creating a New Animation from Scratch

```
STEP 1: Design frames
  ├─ Draw each frame as a separate PNG (use any pixel art tool)
  │  Recommended: Piskel (piskelapp.com), Aseprite, or GIMP
  │  Canvas size: 32×32, 48×48, or 64×64 (keep ≤ 64×64 for CoreV2)
  │  Mode: 1-bit (black and white only — monochrome OLED)
  └─ Export each frame as PNG

STEP 2: Convert to C array
  ├─ Option A: Wokwi Animator (limited to Icons8 library)
  ├─ Option B: image2cpp (javl.github.io/image2cpp)
  │     Upload PNG → set "Vertical - 1 bit per pixel"
  │     → Generate code → copy PROGMEM array
  └─ Option C: LCD Image Converter (desktop, more control)

STEP 3: Integrate into project
  ├─ Create Animations.h / Animations.cpp
  ├─ Paste generated arrays
  ├─ Add frame count, width, height constants
  └─ Use the animation loop pattern from Section 3

STEP 4: Test on hardware
  ├─ Upload to CoreV2
  ├─ Check: no flicker, smooth timing, correct position
  └─ Tweak FRAME_DELAY_MS until it looks right
```

## 12. Troubleshooting & Common Mistakes

### Symptom → Diagnosis → Fix

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| **Animation is garbled / noise** | Wrong byte orientation (horizontal vs vertical) | Ensure converter uses **Vertical - 1 bit per pixel** mode |
| **Image shifted left 2px** | Using SSD1306 code on SH1106 panel | Switch `OledPanelType` or add 2px offset: `x += 2` |
| **Animation flickers badly** | Calling `clearDisplay()` + `display()` every frame | Use `fillRect` partial clear. See Section 3. |
| **Animation too slow / laggy** | Frame delay too low for I²C bandwidth | Increase `FRAME_DELAY_MS`. See frame rate calculator (Section 7). |
| **ESP32 crashes / reboots** | Bitmap array too large for stack | Move large arrays to global scope or use `static`. Never declare >1 KB arrays inside functions. |
| **compilation error: array too large** | Exceeded section size limit | Split animation into multiple files or use PSRAM with `heap_caps_malloc()` |
| **Image is inverted (black ↔ white)** | Bits are flipped | Invert bytes: `~byte` or toggle `SSD1306_WHITE` ↔ `SSD1306_BLACK` |
| **Animation stops after one loop** | Frame counter not resetting | Check `if (currentFrame >= ANIM_FRAMES) currentFrame = 0;` |
| **Memory corruption / weird behavior** | Using GPIO22–37 (reserved for PSRAM) | These are the octal PSRAM pins — **NEVER use them** |
| **Wokwi code won't compile** | Missing `#include <avr/pgmspace.h>` | For ESP32, add at top: `#include <pgmspace.h>` or define `PROGMEM` as empty |

### Quick Checklist Before Asking for Help

```
□ Is the I²C address correct? (0x3C for SSD1306, try 0x3D)
□ Is the Wire.begin() called with GPIO8 (SDA) and GPIO9 (SCL)?
□ Is display.begin(SSD1306_SWITCHCAPVCC, addr) called in setup()?
□ Is FRAME_DELAY_MS reasonable? (try 100 ms first, then tune)
□ Are bitmap arrays in PROGMEM and in global scope (not stack)?
□ Is the animation size ≤ 64×64? (Larger causes I²C slowdown)
□ Are you calling display.display() EXACTLY once per frame? (not in loops)
□ Are you using fillRect to clear instead of clearDisplay?
```

## Bonus: Complete AnimationPlayer Class

A reusable class that handles frame timing, looping, ping-pong, and positioning for any Wokwi-generated animation:

```cpp
// AnimationPlayer.h — drop-in animation engine for CoreV2 projects
#pragma once
#include <Adafruit_GFX.h>

enum class LoopMode : uint8_t {
  ONCE,         // Play once, stop on last frame
  LOOP,         // Continuous loop
  PING_PONG,    // Play forward then backward
  BOUNCE        // Like LOOP but reverse direction each cycle
};

class AnimationPlayer {
public:
  // Configure with a Wokwi-generated animation
  void begin(const unsigned char* frameData,
             uint8_t frameCount, uint8_t width, uint8_t height,
             uint16_t bytesPerFrame, uint32_t frameDelayMs,
             LoopMode mode = LoopMode::LOOP) {
    _frames     = frameData;
    _numFrames  = frameCount;
    _w          = width;
    _h          = height;
    _bytesPerFrame = bytesPerFrame;
    _delayMs    = frameDelayMs;
    _mode       = mode;
    _currentFrame = 0;
    _direction  = 1;
    _playing    = true;
    _lastTime   = 0;
  }

  // Call every frame — returns true if a new frame was drawn
  bool update(Adafruit_SSD1306& display, int16_t x, int16_t y) {
    if (!_playing) return false;

    uint32_t now = millis();
    if (now - _lastTime < _delayMs) return false;
    _lastTime = now;

    // Clear previous frame area
    display.fillRect(x, y, _w, _h, SSD1306_BLACK);

    // Draw current frame
    const uint8_t* frame = _frames + (_currentFrame * _bytesPerFrame);
    display.drawBitmap(x, y, frame, _w, _h, SSD1306_WHITE);

    // Advance
    _currentFrame += _direction;

    // Handle loop modes
    if (_currentFrame >= _numFrames) {
      switch (_mode) {
        case LoopMode::ONCE:
          _currentFrame = _numFrames - 1;
          _playing = false;
          break;
        case LoopMode::LOOP:
          _currentFrame = 0;
          break;
        case LoopMode::BOUNCE:
          _currentFrame = 0;
          break;
        case LoopMode::PING_PONG:
          _currentFrame = _numFrames - 2;
          _direction = -1;
          break;
      }
    } else if (_currentFrame < 0 && _mode == LoopMode::PING_PONG) {
      _currentFrame = 1;
      _direction = 1;
    }

    return true;
  }

  void play()  { _playing = true; }
  void pause() { _playing = false; }
  void reset() { _currentFrame = 0; _direction = 1; _playing = true; }
  bool isPlaying() const { return _playing; }
  uint8_t currentFrame() const { return _currentFrame; }

private:
  const uint8_t* _frames;
  uint8_t  _numFrames, _w, _h, _currentFrame;
  int8_t   _direction;
  uint16_t _bytesPerFrame;
  uint32_t _delayMs, _lastTime;
  LoopMode _mode;
  bool     _playing;
};
```

**Usage:**
```cpp
AnimationPlayer heartAnim;

void setup() {
  // ... display init ...
  heartAnim.begin(&anim_heart_beat[0][0],  // pointer to first byte
                  20,    // 20 frames
                  32, 32, // 32×32 pixels
                  128,    // 128 bytes per frame
                  80,     // 80 ms between frames (~12.5 FPS)
                  LoopMode::PING_PONG);
}

void loop() {
  if (heartAnim.update(display, 48, 16)) {
    display.display();  // Only push when a new frame was drawn
  }
}
```
